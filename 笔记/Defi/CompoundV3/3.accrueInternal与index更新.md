# 3. accrueInternal 与 index 更新

# 一、`accrueInternal()`

在用户交互发生时，把协议的 base 存款利息和借款利息按“从上次 accrual 到现在经过的时间”统一推进到最新状态。源码中它会计算 `timeElapsed`，更新 `baseSupplyIndex` 和 `baseBorrowIndex`，并同步更新奖励 tracking index 与 `lastAccrualTime`。

## accrueInternal 直接更新的核心状态

* `baseBorrowIndex`
* `baseSupplyIndex`
* `trackingBorrowIndex`
* `trackingSupplyIndex`
* `lastAccrualTime`

## accrueInternal 间接影响的结果

通过新的 index，使得

* `totalBorrowBase` 投影出的真实总借款变大
* `totalSupplyBase` 投影出的真实总存款变大

也就是说：

* `totalBorrowBase/totalSupplyBase` 作为 principal 总量，不一定在 accrual 时直接改
* 但它们对应的 **present value** 会因为 index 更新而变化

## 和 V2 的联系与区别

* 都是“本金不频繁改，靠 index 反映利息”
* 但 V3 把 base 头寸统一成一个 signed principal，更集中

## 源码

```solidity
function accrueInternal() internal {
        uint40 now_ = getNowInternal(); //查询现在时间
        uint256 timeElapsed = uint256(now_ - lastAccrualTime); //算离上次结息过了多久
        if (timeElapsed > 0) {
            (baseSupplyIndex, baseBorrowIndex) = accruedInterestIndices(timeElapsed);
            if (totalSupplyBase >= baseMinForRewards) {
                trackingSupplyIndex += safe64(divBaseWei(baseTrackingSupplySpeed * timeElapsed, totalSupplyBase)); //顺便更新奖励指数
            }
            if (totalBorrowBase >= baseMinForRewards) {
                trackingBorrowIndex += safe64(divBaseWei(baseTrackingBorrowSpeed * timeElapsed, totalBorrowBase)); //顺便更新奖励指数
            }
            lastAccrualTime = now_; //把 lastAccrualTime 改成现在
        }
    }
```

### `getNowInternal()`

本质就是取当前区块时间戳，只不过源码额外做了一个范围保护，然后压成 `uint40`

```solidity
function getNowInternal() internal view virtual returns (uint40) {
        if (block.timestamp >= 2 ** 40) revert TimestampTooLarge();
        return uint40(block.timestamp);
    }
```

这说明 `accrueInternal()` 的推进依据就是**时间流逝**，不是区块数，也不是用户数

### `timeElapsed`

```solidity
uint timeElapsed = uint256(now_ - lastAccrualTime);
```

协议不在乎“你是谁触发了这次操作”，只在乎“距离上一次 accrual 已经过了几秒”。

如果 `timeElapsed == 0`，说明这次和上次 accrual 在同一个时间点，那就没必要重新算利息，所以源码直接什么也不做。只有 `timeElapsed > 0` 才会继续往下更新。

可以把这一步理解成：

* V3 不是连续实时改状态，它是“**懒更新**”
* 谁来操作不重要，重要的是**时间已经过去了多少**

### `accruedInterestIndices(timeElapsed)`

```solidity
(baseSupplyIndex, baseBorrowIndex) = accruedInterestIndices(timeElapsed);
```

```solidity
 function accruedInterestIndices(uint256 timeElapsed) internal view returns (uint64, uint64) {
        uint64 baseSupplyIndex_ = baseSupplyIndex;
        uint64 baseBorrowIndex_ = baseBorrowIndex;
        if (timeElapsed > 0) {
            uint256 utilization = getUtilization();
            uint256 supplyRate = getSupplyRate(utilization);
            uint256 borrowRate = getBorrowRate(utilization);
            baseSupplyIndex_ += safe64(mulFactor(baseSupplyIndex_, supplyRate * timeElapsed));
            baseBorrowIndex_ += safe64(mulFactor(baseBorrowIndex_, borrowRate * timeElapsed));
        }
        return (baseSupplyIndex_, baseBorrowIndex_);
    }
```

#### `getUtilization()`

用当前的`baseSupplyIndex/baseBorrowIndex` 和全局 principal 投影出真实总存款、真实总借款，算出 utilization。

```solidity
function getUtilization() public view override returns (uint256) {
        uint256 totalSupply_ = presentValueSupply(baseSupplyIndex, totalSupplyBase); //投影成当前真实总存款
        uint256 totalBorrow_ = presentValueBorrow(baseBorrowIndex, totalBorrowBase); //投影成当前真实总借款
        if (totalSupply_ == 0) {
            return 0;
        } else {
            return totalBorrow_ * FACTOR_SCALE / totalSupply_;
        }
    }
```

#### `getSupplyRate()` 和 `getBorrowRate()`

把 utilization 喂给 `getSupplyRate()` 和 `getBorrowRate()`，得到当前每秒利率。

```solidity
function getSupplyRate(uint256 utilization) public view override returns (uint64) {
        //分段函数 利用率小于等于 kink，一条低斜率
        if (utilization <= supplyKink) {
            // interestRateBase + interestRateSlopeLow * utilization
            return safe64(supplyPerSecondInterestRateBase + mulFactor(supplyPerSecondInterestRateSlopeLow, utilization));
        } else {
            //大于 kink，切到高斜率部分
        // interestRateBase + interestRateSlopeLow * kink + interestRateSlopeHigh * (utilization - kink)
            return safe64(
                supplyPerSecondInterestRateBase + mulFactor(supplyPerSecondInterestRateSlopeLow, supplyKink)
                    + mulFactor(supplyPerSecondInterestRateSlopeHigh, (utilization - supplyKink))
            );
        }
    }
```

#### <font style="background-color:#FBF5CB;">更新指数</font><code><font style="background-color:#FBF5CB;">baseSupplyIndex</font></code><font style="background-color:#FBF5CB;"> 和 </font><code><font style="background-color:#FBF5CB;">baseBorrowIndex</font></code>

```solidity
baseSupplyIndex_ += safe64(mulFactor(baseSupplyIndex_, supplyRate * timeElapsed));
baseBorrowIndex_ += safe64(mulFactor(baseBorrowIndex_, borrowRate * timeElapsed));
```

即，**指数的增长 = 原指数 × 利率 × 时间 ：**

* **newSupplyIndex** = oldSupplyIndex + oldSupplyIndex × supplyRate × Δt
* **newBorrowIndex** = oldBorrowIndex + oldBorrowIndex × borrowRate × Δt

只是源码里还通过 `mulFactor(...)` 做了比例缩放处理。这说明 index 本身也是按比例增长的。

### 写回新index、更新奖励指数

```solidity
(baseSupplyIndex, baseBorrowIndex) = accruedInterestIndices(timeElapsed);
```

也就是把刚才算出来的两个新 index，写回全局状态。\
这一步完成后，协议里的“利息时间轴”就正式推进到了当前时刻。

然后它还会**更新奖励指数**：

```solidity
if (totalSupplyBase >= baseMinForRewards) {
    trackingSupplyIndex += ...
}
if (totalBorrowBase >= baseMinForRewards) {
    trackingBorrowIndex += ...
}
```

### `lastAccrualTime`

```solidity
lastAccrualTime = now_;
```

也就是告诉系统：我已经结息到这个时间点了，下次从这里继续往后算。

# 二、principal的逻辑原理

大白话，我的理解，用户存钱或者借钱的时候，就按当时的index压缩成一个principal，只要没有存取钱这个principal就不会变，变的只是index；

需要真实余额的时候，就按需要时的index来解压缩得到真实余额；

比如又要存钱，存amount，就按照这回要存钱时的index解压缩出真实余额，然后这个真实余额+amount以后，再按照这时的index压缩回去成一个新的principal，依旧是没有存取钱就不会变。

既然 principal 不自动变，那用户每次操作时，到底是怎么把“旧 principal + 新操作”重新变成“新 principal”的？

答案就是这三个函数分工完成的：

1. `presentValue(...)`\
   先把旧 principal 还原成**当前真实余额**
2. `principalValue(...)`\
   再把“操作后的真实余额”压回**新的 principal**
3. `updateBasePrincipal(...)`\
   最后把新的 principal 写回用户状态，并处理 base tracking 奖励相关更新。源码里这几个函数都在 `Comet.sol` 的主逻辑中。

## `presentValue()`

### 源码

```solidity
function presentValue(int104 principalValue_) internal view returns (int256) {
        if (principalValue_ >= 0) {
            //头寸>0,用baseSupplyIndex
            return signed256(presentValueSupply(baseSupplyIndex, uint104(principalValue_)));
        } else {
            //头寸<0,用baseBorrowIndex
            return -signed256(presentValueBorrow(baseBorrowIndex, uint104(-principalValue_)));
        }
    }
```

### 头寸 > 0 --> 用 base<font style="background-color:#FBDFEF;">Supply</font>Index

```solidity
function principalValueSupply(uint64 baseSupplyIndex_, uint256 presentValue_) internal pure returns (uint104) {
        //向下取整,对协议更保守，不会因为舍入给存款人多算资产
        return safe104((presentValue_ * BASE_INDEX_SCALE) / baseSupplyIndex_);
    }
```

`present supply = principal × baseSupplyIndex / BASE_INDEX_SCALE`

这里是**向下取整**,对协议更保守，不会因为舍入给存款人多算资产

### 头寸 < 0 --> 用 base<font style="background-color:#FBDFEF;">Borrow</font>Index

```solidity
function presentValueBorrow(uint64 baseBorrowIndex_, uint104 principalValue_) internal pure returns (uint256) {
  return uint256(principalValue_) * baseBorrowIndex_ / BASE_INDEX_SCALE;
}
```

`present borrow = principal × baseBorrowIndex / BASE_INDEX_SCALE`

* principal 是压缩后的债务
* borrow index 越涨，真实欠的钱越多

注意看\*\*(a + b - 1) / b\*\* ,经典向上取整,即协议不会因为舍入而少记借款人的债务

### 为什么 borrow 和 supply 分开？

因为协议通常让：<font style="background-color:#CEF5F7;"></font>

* 借款利率更高
* 存款利率更低

所以：

* `baseBorrowIndex` 增长更快
* `baseSupplyIndex` 增长更慢

同样 100 principal：

* 借款人的真实债务涨得更快
* 存款人的真实资产涨得更慢     <font style="background-color:#FBF5CB;">=>这就是协议利差来源之一  </font>

## `principalValue()`

```solidity
 function principalValue(int256 presentValue_) internal view returns (int104) {
        if (presentValue_ >= 0) {
            //>0
            return signed104(principalValueSupply(baseSupplyIndex, uint256(presentValue_)));
        } else {
            //<0
            return -signed104(principalValueBorrow(baseBorrowIndex, uint256(-presentValue_)));
        }
    }
```

\*\* principal = 真实余额 × scale / index  \*\*

最后返回是**uint104**，因为用户状态里最终存的是\*\* signed principal  \*\*

## `updateBasePrincipal()`

\*\*先按旧头寸把历史奖励结清，再把用户 principal 更新成新值，并把奖励起点切到新头寸对应的 tracking index。  \*\*

V3 把 base 头寸统一成一个 signed principal，更集中

* 存款通常表现为正 principal
* 借款通常表现为负 principal
* 用户操作后，最终由该函数完成落账

### 源码

```solidity
//把用户新的 base 头寸写回去，并更新与奖励/跟踪相关的状态
    function updateBasePrincipal(address account, UserBasic memory basic, int104 principalNew) internal {
        int104 principal = basic.principal; //取用户操作之前的旧 principal
        basic.principal = principalNew; //先把新 principal 写到内存里的 basic，还没写回 storage

        //旧 principal >= 0 的情况
        if (principal >= 0) {
            uint256 indexDelta = uint256(trackingSupplyIndex - basic.baseTrackingIndex);
            //从用户上次记录的 tracking index，到当前全局 trackingSupplyIndex，增长了多少
            basic.baseTrackingAccrued += safe64(
                uint104(principal) * indexDelta / trackingIndexScale / accrualDescaleFactor
            ); //奖励通常和你的头寸大小成比例,头寸越大，同样的 index 增量下，你应得奖励越多
            //把“principal × 奖励指数增量”从缩放后的整数世界还原到实际奖励单位

            //旧 principal < 0 的情况
        } else {
            uint256 indexDelta = uint256(trackingBorrowIndex - basic.baseTrackingIndex);
            basic.baseTrackingAccrued += safe64(
                uint104(-principal) * indexDelta / trackingIndexScale / accrualDescaleFactor
            ); //借款人如果也有借款奖励，就按“借款绝对值 × 借款 tracking 增量”累计
        }

        if (principalNew >= 0) {
            basic.baseTrackingIndex = trackingSupplyIndex; //把这次结算出来的奖励，累加进用户已累计奖励里。
        } else {
            basic.baseTrackingIndex = trackingBorrowIndex;
        }

        userBasic[account] = basic; //真正把新principal写回链上
    }
```

* 旧 principal >= 0：按 `trackingSupplyIndex` 结算历史奖励
* 旧 principal < 0：按 `trackingBorrowIndex` 结算历史奖励
* 新 principal >= 0：新的 tracking 起点设为 `trackingSupplyIndex`
* 新 principal < 0：新的 tracking 起点设为 `trackingBorrowIndex`

## 难点情况：借款人还款后变成存款人

假设吉伊：

* 原来`principal` = -100，说明是借款人

当前真实债务因为 borrow index 增长，已经变成：

* `presentValue(-100) = -120`

现在吉伊还了 150

### 第一步：旧头寸展开

旧真实头寸是：

```plain
-120
```

### 第二步：加上本次操作

因为还款 150，相当于 base 头寸增加 150：

```plain
-120 + 150 = +30
```

注意，这时候吉伊已经从负债状态翻成正资产状态了。

### 第三步：压回 principal

调用：

```plain
principalValue(+30)
```

因为是正数，走 supply 方向，得到正的

### 第四步：updateBasePrincipal

这里特别精彩：

#### 先按旧 principal = -100

结清过去这一段 **借款奖励**

所以走的是：

```plain
trackingBorrowIndex - basic.baseTrackingIndex
```

#### 然后发现 `principalNew >= 0`

所以把新的 `baseTrackingIndex` 切成：

```plain
trackingSupplyIndex
```

这表示：

过去吉伊是借款人，那段奖励按借款赛道结\
现在吉伊变存款人了，以后从存款赛道继续跑

这就是整个切换逻辑。
