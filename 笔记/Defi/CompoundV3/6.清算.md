# 6. 清算

# `absorb()`

`absorb` 是协议接管，不是清算人直接拿走资产。清算执行者 `absorber` 更像是触发者/执行者，\
而坏账和 collateral 首先是被协议接到自己账上。

```solidity
function absorb(address absorber, address[] calldata accounts) external override {
  if (isAbsorbPaused()) revert Paused(); //确认 absorb 没被暂停

  uint256 startGas = gasleft(); //记录初始 gas
  accrueInternal(); //推进利息时间轴,更新index
  //遍历一组账户，逐个 absorb
  for (uint256 i = 0; i < accounts.length;) {
    absorbInternal(absorber, accounts[i]);
    unchecked {
      i++;
    }
  }
  uint256 gasUsed = startGas - gasleft(); //统计 gas 使用量

  // Note: liquidator points are an imperfect tool for governance,
  //  to be used while evaluating strategies for incentivizing absorption.
  // Using gas price instead of base fee would more accurately reflect spend,
  //  but is also subject to abuse if refunds were to be given automatically.

  //给 absorber 记点数
  LiquidatorPoints memory points = liquidatorPoints[absorber];
  points.numAbsorbs++;
  points.numAbsorbed += safe64(accounts.length);
  points.approxSpend += safe128(gasUsed * block.basefee);
  liquidatorPoints[absorber] = points;
}
```

## 账户逐个absorbinternal

```solidity
for (uint256 i = 0; i < accounts.length;) {
    absorbInternal(absorber, accounts[i]);
    unchecked {
        i++;
    }
}
```

说明：

* 一次 absorb 调用可以处理多个账户
* 每个账户独立走一遍 `absorbInternal()`

真正的“吸收坏账”都在 `absorbInternal()`

## 给absorber记点数

```solidity
 //给 absorber 记点数
  LiquidatorPoints memory points = liquidatorPoints[absorber];
  points.numAbsorbs++;
  points.numAbsorbed += safe64(accounts.length);
  points.approxSpend += safe128(gasUsed * block.basefee);
  liquidatorPoints[absorber] = points;
```

这里不是直接给代币奖励，而是在账上记：

* 你做了多少次 absorb
* 吸收了多少个账户
* 大概花了多少 basefee 成本

注释里也说了，这是一种 imperfect tool，不是特别精确的自动补偿机制，更像治理后续可能用来参考的“清算贡献记录”。

# `absorbInternal()`

```solidity
 function absorbInternal(address absorber, address account) internal {
        if (!isLiquidatable(account)) revert NotLiquidatable(); //先检查是否可清算

        //读取账户旧状态
        UserBasic memory accountUser = userBasic[account];
        int104 oldPrincipal = accountUser.principal; //用户旧的 base signed principal
        int256 oldBalance = presentValue(oldPrincipal); //这是用户当前真实 base 头寸,负债是负的
        uint16 assetsIn = accountUser.assetsIn; //用户有哪些 collateral 当前纳入风控

        uint256 basePrice = getPrice(baseTokenPriceFeed); //拿到 base 价格
        uint256 deltaValue = 0; //初始化 deltaValue:所有被没收抵押物，按清算规则折算出来的总价值

        //遍历所有纳入风控的 collateral，并全部没收
        for (uint8 i = 0; i < numAssets;) {
            if (isInAsset(assetsIn, i)) {
                //判断这个资产是否在 assetsIn
                AssetInfo memory assetInfo = getAssetInfo(i);
                address asset = assetInfo.asset; //读取资产信息,拿到collateral的配置
                uint128 seizeAmount = userCollateral[account][asset].balance; //把用户该资产的 collateral 余额取出来
                userCollateral[account][asset].balance = 0; //直接把用户这项 collateral 清零
                totalsCollateral[asset].totalSupplyAsset -= seizeAmount; //全局 collateral 总量也扣掉
                //把没收的 collateral 换算成价值
                uint256 value = mulPrice(seizeAmount, getPrice(assetInfo.priceFeed), assetInfo.scale); //抵押物数量 × 价格
                deltaValue += mulFactor(value, assetInfo.liquidationFactor); //按 liquidationFactor 折算后，再作为可用于处理债务的价值

                emit AbsorbCollateral(absorber, account, asset, seizeAmount, value);
            }
            unchecked {
                i++;
            }
        }

        uint256 deltaBalance = divPrice(deltaValue, basePrice, uint64(baseScale)); //把 deltaValue 从价值维度换成 base 数量维度
        int256 newBalance = oldBalance + signed256(deltaBalance); //用这部分 base 等价物去冲用户旧债务
        // New balance will not be negative, all excess debt absorbed by reserves:清算后，用户不会保留负债。
        if (newBalance < 0) {
            newBalance = 0; //如果还没填平，就直接把剩余坏账抹到 0
        }

        int104 newPrincipal = principalValue(newBalance); //把新的 newBalance 压回 principal
        //如果 absorb 过头导致 newBalance > 0，那也有可能变成正 principal，表示用户甚至留下少量净存款
        updateBasePrincipal(account, accountUser, newPrincipal);

        // reset assetsIn
        userBasic[account].assetsIn = 0; //清空 assetsIn

        //拆分 principal 变化，更新全局供借 principal
        (uint104 repayAmount, uint104 supplyAmount) = repayAndSupplyAmount(oldPrincipal, newPrincipal);

        // Reserves are decreased by increasing total supply and decreasing borrows
        //  the amount of debt repaid by reserves is `newBalance - oldBalance`
        totalSupplyBase += supplyAmount;
        totalBorrowBase -= repayAmount;

        //记录协议为这个账户实际“付出了多少 base”
        uint256 basePaidOut = unsigned256(newBalance - oldBalance);
        uint256 valueOfBasePaidOut = mulPrice(basePaidOut, basePrice, uint64(baseScale));
        emit AbsorbDebt(absorber, account, basePaidOut, valueOfBasePaidOut);

        //如果 absorb 后用户成了正 principal，发 mint 型 Transfer 事件
        if (newPrincipal > 0) {
            emit Transfer(address(0), account, presentValueSupply(baseSupplyIndex, unsigned104(newPrincipal)));
        }
    }
```

1. 先确认账户真的可清算
2. 读取账户旧 base 债务状态
3. 把账户所有纳入风控的 collateral 全部没收清零
4. 把这些 collateral 按清算折扣规则折算成“能抵多少 base 债”
5. 用这部分价值去冲减用户 base 负债
6. 如果还不够还清，剩余坏账由协议储备兜底，所以用户最终不会保留负债
7. 更新用户 principal、全局供借账本、清空 assetsIn

## 检查是否可清算

```solidity
if (!isLiquidatable(account)) revert NotLiquidatable();
```

只有已经跌破清算线的账户，才允许被 absorb

## 读取账户旧状态

### `oldPrincipal`

用户旧的 base signed principal

可能是负数，实际上这里能进清算几乎必然是负数，因为有 base 借款。

### `oldBalance`

```solidity
presentValue(oldPrincipal)
```

这是用户当前真实 base 头寸。因为是坏账账户，所以这里应该是负数，代表真实 base 债务

### `assetsIn`

用户有哪些 collateral 当前纳入风控。

后面要把这些抵押物一项项全部吸走。

## 拿到 base 价格，并初始化`deltaValue`

```solidity
uint256 basePrice = getPrice(baseTokenPriceFeed); //拿到 base 价格
uint256 deltaValue = 0; //初始化 deltaValue:所有被没收抵押物，按清算规则折算出来的总价值
```

`deltaValue` 是一个累计变量,累计的不是base数量,而是\*\*所有被没收抵押物，按清算规则折算出来的总价值 \*\*

先是价值维度，不是 base token 数量维度

```solidity
    }

```

### (1) 先判断这个资产是否在 `assetsIn`

```solidity
if (isInAsset(assetsIn, i))
```

只有参与风控的资产才处理

### (2) 读取资产信息

```solidity
AssetInfo memory assetInfo = getAssetInfo(i);
address asset = assetInfo.asset;
```

拿到 collateral 的配置

### (3) 把用户该资产的 collateral 余额取出来

```solidity
uint128 seizeAmount = userCollateral[account][asset].balance;
```

这次要被协议没收的该资产数量

### (4) 直接把用户这项 collateral 清零

```solidity
userCollateral[account][asset].balance = 0;
```

**协议把这项抵押物整个接管了。**

不是留一部分，不是交易一部分，而是这里直接清零。

### (5) 全局 collateral 总量也扣掉

```solidity
totalsCollateral[asset].totalSupplyAsset -= seizeAmount;
```

说明全局账本里，这部分 collateral 也不再算作“用户供应的 collateral”。因为已经被协议接管。

### (6) 把没收的 collateral 换算成价值

```solidity
uint256 value = mulPrice(seizeAmount, getPrice(assetInfo.priceFeed), assetInfo.scale);
```

**抵押物数量 × 价格**

得到原始价值

### (7) 再乘 `liquidationFactor`，累计进 `deltaValue`

```solidity
deltaValue += mulFactor(value, assetInfo.liquidationFactor);
```

**deltaValue += mulFactor(value, assetInfo.****<font style="background-color:#FBDFEF;">liquidationFactor</font>****);**

#### <code>**liquidationFactor**</code>

被没收的 collateral，最终折算多少价值来冲减债务。

* 为什么会有这个factor,而不是全部base价值都拿去抵债?

因为清算本质不是说 collateral 原始价值和债务一模一样精确交换。

协议会用一套风险折算规则去处理：

* 资产波动风险
* 执行风险
* 协议储备安全边际

所以：

* collateral 全部被接管
* 但能拿来冲债的有效价值，是折算后的 `deltaValue`

***

注意这一段结束后，`deltaValue`依旧代表价值维度，就是这个坏账户被没收的全部collateral按清算折算规则换算后，总共能贡献多少价值来填补base债务

## `deltaValue`转换到 base 数量维度

```solidity
uint256 deltaBalance = divPrice(deltaValue, basePrice, uint64(baseScale));
```

**价值 ÷ base价格, 即**

**相当于多少 base token 数量**

于是 `deltaBalance` 表示：

这些被没收 collateral，折算后相当于能补多少 base 债务

## 用这部分 base 等价物去冲用户旧债务

```solidity
int256 newBalance = oldBalance + signed256(deltaBalance);
```

* `oldBalance` 是负数，比如 `-1200`
* `deltaBalance` 是正数，比如 `+800`

所以：

```solidity
newBalance = -1200 + 800 = -400
```

意思就是：

**原本欠 1200，现在 collateral 折算后替你填了 800，还剩 400 坑。**

所以虽然写的是加号，本质上是用 collateral 价值去冲减负债

## 如果还没填平，就直接把剩余坏账抹到 0

```solidity
if (newBalance < 0) {
    newBalance = 0;
}
```

<font style="background-color:#FBDFEF;">清算后，用户不会保留负债</font>

如果 collateral 折算后还不够抵掉全部债务，剩下的坏账由**协议储备**承担。

## 把新的 `newBalance` 压回 principal

```solidity
int104 newPrincipal = principalValue(newBalance);
updateBasePrincipal(account, accountUser, newPrincipal);
```

* `oldBalance` 是旧真实头寸
* absorb 后算出 `newBalance`
* 再压回 `newPrincipal`
* 写回用户状态

如果前面 `newBalance` 被钳成了 0，

那这里通常就意味着：

```solidity
newPrincipal = 0
```

即用户的 base 债务被清空。

如果 absorb 过头导致 `newBalance > 0`，那也有可能变成正 principal，表示**用户甚至留下少量净存款。**

虽然这不是最常见，但源码是允许的，所以后面有 `if (newPrincipal > 0)` 的事件处理。

***

依旧要`updateBasePrincipal()` 是因为即使是清算，用户 base 头寸也是一次“旧头寸 -> 新头寸”的状态转移。即：

* 用旧 principal 结清 tracking
* 写入新 principal
* 更新 tracking 起点

## 清空 `assetsIn`

```solidity
userBasic[account].assetsIn = 0;
```

这个用户现在已经没有任何参与风控的抵押物了，所以 `assetsIn` 也该归零。

## 拆分 principal 变化，更新全局供借 principal

```solidity
(uint104 repayAmount, uint104 supplyAmount) = repayAndSupplyAmount(oldPrincipal, newPrincipal);

totalSupplyBase += supplyAmount;
totalBorrowBase -= repayAmount;
```

## 记录协议为这个账户实际“付出了多少 base”

```solidity
uint256 basePaidOut = unsigned256(newBalance - oldBalance);
uint256 valueOfBasePaidOut = mulPrice(basePaidOut, basePrice, uint64(baseScale));
emit AbsorbDebt(absorber, account, basePaidOut, valueOfBasePaidOut);
```

这里：

* `oldBalance` 是负的
* `newBalance` 比它更大（更接近 0，甚至可能 >0）

所以：

```solidity
newBalance - oldBalance
```

是一个正数。

这表示：

**协议总共替这个账户垫了多少 base 头寸改善量**

比如：

* `oldBalance = -1200`
* `newBalance = 0`

那：

```solidity
basePaidOut = 1200
```

可以理解为协议为了把这个账户从 `-1200` 救到 `0`，一共处理了 1200 的 base 债务规模。

然后再乘价格，得到价值维度，发事件记录。

## 如果 absorb 后用户成了正 principal，发 mint 型 Transfer 事件

```solidity
if (newPrincipal > 0) {
  emit Transfer(address(0), account, presentValueSupply(baseSupplyIndex, unsigned104(newPrincipal)));
}
```

这和前面在 `supplyBase(...)` 里看到的逻辑一脉相承。

如果 absorb 后用户不但没债了，反而还剩一点正的 base 头寸，那协议会发一个对应的 `Transfer(address(0), ...)` 事件。

说明账户 absorb 完之后留下了正向 supply 余额。

# 协议如何补reverses

## `buyCollateral()`

```solidity
 function buyCollateral(address asset, uint256 minAmount, uint256 baseAmount, address recipient)
        external
        override
        nonReentrant
    {
        if (isBuyPaused()) revert Paused(); //暂停检查

        int256 reserves = getReserves();
        if (reserves >= 0 && uint256(reserves) >= targetReserves) revert NotForSale(); //先看 reserves 是否真的不足

        // Note: Re-entrancy can skip the reserves check above on a second buyCollateral call.
        //用户先把 base token 转进协议
        //后续逻辑用的是 doTransferIn 返回的实际数量
        baseAmount = doTransferIn(baseToken, msg.sender, baseAmount);

        //算这笔 base 能买多少 collateral
        uint256 collateralAmount = quoteCollateral(asset, baseAmount);
        if (collateralAmount < minAmount) revert TooMuchSlippage(); //滑点保护
        if (collateralAmount > getCollateralReserves(asset)) revert InsufficientReserves(); //协议手里必须真有这么多 collateral reserves

        // Note: Pre-transfer hook can re-enter buyCollateral with a stale collateral ERC20 balance.
        //  Assets should not be listed which allow re-entry from pre-transfer now, as too much collateral could be bought.
        //  This is also a problem if quoteCollateral derives its discount from the collateral ERC20 balance.
        doTransferOut(asset, recipient, safe128(collateralAmount)); //协议把 collateral 发给买家指定地址

        emit BuyCollateral(msg.sender, asset, baseAmount, collateralAmount); //发事件
    }
```

## `quoteCollateral()`

```solidity
 //定价函数
    //协议卖 collateral 时，折扣到底怎么来的
    function quoteCollateral(address asset, uint256 baseAmount) public view override returns (uint256) {
        //拿到资产信息与价格
        AssetInfo memory assetInfo = getAssetInfoByAddress(asset); //目标 collateral 的配置
        uint256 assetPrice = getPrice(assetInfo.priceFeed); //它当前价格
        // Store front discount is derived from the collateral asset's liquidationFactor and storeFrontPriceFactor
        // discount = storeFrontPriceFactor * (1e18 - liquidationFactor)
        //算折扣系数 discountFactor
        uint256 discountFactor = mulFactor(storeFrontPriceFactor, FACTOR_SCALE - assetInfo.liquidationFactor);
        uint256 assetPriceDiscounted = mulFactor(assetPrice, FACTOR_SCALE - discountFactor); //算出折扣后的 collateral 价格
        uint256 basePrice = getPrice(baseTokenPriceFeed); //拿 base 价格
        // # of collateral assets
        // = (TotalValueOfBaseAmount / DiscountedPriceOfCollateralAsset) * assetScale
        // = ((basePrice * baseAmount / baseScale) / assetPriceDiscounted) * assetScale
        return basePrice * baseAmount * assetInfo.scale / assetPriceDiscounted / baseScale;
    }
```

### 折扣系数`discountFactor`

```solidity
uint256 discountFactor = mulFactor(storeFrontPriceFactor, FACTOR_SCALE - assetInfo.liquidationFactor);
```

#### 折价空间

FACTOR\_SCALE - assetInfo.liquidationFactor , 即

```solidity
(1 - liquidationFactor)
```

如果 `liquidationFactor` 越接近 1，那 `(1 - liquidationFactor)` 越小，折扣空间越小。

如果 `liquidationFactor` 更低，那折扣空间更大。

### 折扣力度系数`storeFrontPriceFactor`

就是不一定把全部折价空间全给买家,这个`storeFrontPriceFactor`就好像一个折扣力度吧,乘完这个就得到最终给买家的价格

```solidity
uint256 assetPriceDiscounted = mulFactor(assetPrice, FACTOR_SCALE - discountFactor); 
//算出折扣后的 collateral 价格
```
