# 5. 风控判断

**账户当前的 base 负债价值，能不能被抵押物的“可计入价值”覆盖住？**

区别只在于：

* `isBorrowCollateralized()` 用的是 **borrowCollateralFactor**
  * 判断：**你现在这笔借款允不允许存在**
  * 用在借款/提款后的准入检查
* `isLiquidatable()` 用的是 **liquidateCollateralFactor**
  * 判断：**你现在是不是已经差到可以被清算了**
  * 用在清算资格判断

## `isBorrowCollateralized()`

先把用户当前 base 负债按价格换成负的价值，再把每个抵押物的“可借价值（价值 × borrowCollateralFactor）”一项项加回来，只要最后总和不小于 0，就说明该账户的借款是被足额抵押的。

```solidity
function isBorrowCollateralized(address account) public view override returns (bool) {
        int104 principal = userBasic[account].principal; //取 principal

        if (principal >= 0) {
            // 当前没有base借款
            return true; // 可以借的 / collateralized 的
            // 连负债都没有，风控上不可能不够抵押
        }

        //以下else
        uint16 assetsIn = userBasic[account].assetsIn; //把你当前有哪些抵押物在参与风控拿出来
        int256 liquidity = signedMulPrice(presentValue(principal), getPrice(baseTokenPriceFeed), uint64(baseScale));
        //（投影出真实base 头寸的价值 * base token 的价格）再精度缩放
        //这里是先把你的抵押值记成负的

        //遍历抵押物，把“可借价值”一项项加回来
        for (uint8 i = 0; i < numAssets;) {
            if (
                isInAsset(assetsIn, i) //这个资产如果余额为 0，就跳过
                //如果余额非 0，就拿出来参与风控计算
            ) {
                if (liquidity >= 0) {
                    return true; //如果前面已经靠之前若干个抵押物把负债“补正了”，那就不用继续遍历了，直接返回 true
                }

                AssetInfo memory asset = getAssetInfo(i); //取这个 collateral 的信息
                //把某个 collateral 的余额换成价值,不是具体多少钱,是统一的价值
                uint256 newAmount =
                    mulPrice(userCollateral[account][asset.asset].balance, getPrice(asset.priceFeed), asset.scale);
                liquidity += signed256(mulFactor(newAmount, asset.borrowCollateralFactor)); //抵押物价值 × borrowCollateralFactor
            }
            unchecked {
                i++;
            }
        }

        return liquidity >= 0;
    }
```

这里

```solidity
 int256 liquidity = signedMulPrice(presentValue(principal), getPrice(baseTokenPriceFeed), uint64(baseScale));
```

* `presentValue(principal)`：把用户当前 principal 投影成当前真实 base 头寸
* `getPrice(baseTokenPriceFeed)`：base token 的价格
* `uint64(baseScale)`：精度缩放

### 风控估值核心

```solidity
AssetInfo memory asset = getAssetInfo(i);
//把某个 collateral 的余额换成价值,不是具体多少钱,是统一的价值
uint256 newAmount =
    mulPrice(userCollateral[account][asset.asset].balance, getPrice(asset.priceFeed), asset.scale);
liquidity += signed256(mulFactor(newAmount, asset.borrowCollateralFactor));
```

1. 先取这个collateral的信息，拿到的是这类抵押物的配置，比如：

* token 地址
* price feed
* scale
* borrow collateral factor
* liquidate collateral factor

2. 把 collateral 数量换算成“原始价值”

```solidity
uint256 newAmount =
    mulPrice(userCollateral[account][asset.asset].balance, getPrice(asset.priceFeed), asset.scale);
```

```solidity
余额 × 价格 ÷ 精度
```

得到这个抵押物当前的价值。

比如：

* 你有 10 个 ETH
* ETH 价格 3000
* 那价值大概就是 30000

3. 最后计算`liquidity`：

```solidity
 抵押物价值 × borrowCollateralFactor
```

它不是把抵押物全价值加进去，而是只加**可借价值**

#### 为什么不按 100% 算？

因为协议需要留安全边际。抵押物会波动，不可能把全部价值都算成可借额度。所以要打折。

这就是 collateral factor 的本质。

## `isLiquidatable()`

与上边`isBorrowCollateralized()`最大的区别就是\*\*用的\*\*\*\*<font style="background-color:#FBDFEF;">factor</font>\*\***不一样**

```solidity
asset.borrowCollateralFactor
```

```solidity
asset.liquidateCollateralFactor
```

### `borrowCollateralFactor`

用于判断：

**你最多允许借到哪里**

它决定借款准入线,通常更保守。

比如某抵押物价值 100，可能只允许算 80 作为借款能力。

### `liquidateCollateralFactor`

用于判断：

**你跌到哪里就可以被清算**

它决定清算线。通常会比 `borrowCollateralFactor` 更大一些，或者说更“宽松”一些，从而形成缓冲区。

#### 为什么要两条线？

因为协议不希望出现这种情况：

* 你刚借出来的一瞬间，只要价格波动一点点，就立刻能被清算

那用户体验太差，也不合理。

所以会设计成：

* \*\*借款准入线更严格,\*\*要借钱时，要求抵押更充足
* \*\*清算线更宽,\*\*已经借出去后，允许你在一定范围内波动，不会刚碰线就清算

这两条线之间的空间，通常就叫：

### buffer / 缓冲区

假设用户有一项抵押物：

* 市值 = 100

设：

* `borrowCollateralFactor = 80%`
* `liquidateCollateralFactor = 90%`

如果你的负债在：

```plain
80 < debt <= 90
```

这时会发生:

`isBorrowCollateralized(...)`不通过,因为已经超过借款准入线了

`isLiquidatable(...)`还不一定能清算,因为还没跌破清算线

这就是缓冲区。
